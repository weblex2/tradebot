# Session Summary – 2026-03-21

## Probleme & Fixes

### 1. Falsche Trade-Cancellations (Sell-Orders)

**Problem:** Sell-Orders wurden fälschlicherweise gecancelt, obwohl Assets im Portfolio vorhanden waren.

**Ursachen:**
- `getPortfolioBreakdown()` lieferte den Gesamt-Bestand inkl. gestakter/gesperrter Coins — Coinbase erlaubt aber nur den `available_balance` zu verkaufen
- Preisdrift zwischen Entscheidungszeitpunkt und Ausführung führte zu minimalen Rundungsfehlern (z.B. €19.28 verfügbar, €19.29 benötigt)
- `getPrices()` lieferte rohe EUR-Float statt Integer-Cents (Faktor 100x falsch)

**Fixes in `CoinbaseService`:**
- `getAvailableBalance(string $currency): ?float` hinzugefügt — nutzt den `available_balance` vom Accounts-Endpoint
- `getPrices()` korrigiert: `(int) round((($bid + $ask) / 2) * 100)` statt rohem Float
- `getPortfolioBreakdown()` nutzt jetzt ebenfalls `available_balance`

**Fix in `TradeExecutor::runChecks()`:**
- Rückgabetyp von `?string` auf `?array` geändert (`[status, reason]`)
- Unterscheidung `cancelled` (Geldmangel) vs. `failed` (Hard Error)
- Sell-Check vergleicht jetzt in Crypto-Einheiten mit 1% Preisdrift-Toleranz:
  ```php
  $neededCrypto = ($decision->amount_usd / 100) / ($priceCents / 100);
  $availableWithTolerance = $availableCrypto * 1.01;
  if ($availableWithTolerance < $neededCrypto) → cancelled
  ```

### 7. Kritischer Fehler: Tests haben Produktions-DB gelöscht

**Was passiert ist:** `RefreshDatabase` Trait führt `migrate:fresh` auf der konfigurierten DB aus — das war die Produktions-DB `trading`. Alle Daten (Articles, Analyses, Executions, Users) wurden gelöscht.

**Sofortmaßnahmen:**
- User wiederhergestellt: `admin@tradebot.local` / `admin123`
- Sources neu geseeded
- Scraper manuell ausgeführt → 141 Articles, 156 Signals wiederhergestellt
- Analyzer manuell ausgeführt → Analysis #16 erstellt

**Dauerhafter Fix:**
- `RefreshDatabase` → `DatabaseTransactions` in allen Tests
- `DatabaseTransactions` wrapped jeden Test in eine Transaktion, die am Ende zurückgerollt wird — die Produktions-DB bleibt unberührt

**`phpunit.xml`:** SQLite-Zeilen wieder auskommentiert (MySQL bleibt Standard)

**Migrationen SQLite-kompatibel gemacht** (für eventuelle zukünftige SQLite-Tests):
- `add_gemini_to_bot_logs_channel.php` — `if (DB::getDriverName() !== 'mysql') return;`
- `add_analyzer_to_bot_logs_channel.php` — dto.
- `change_bot_logs_message_to_text.php` — dto.
- `create_bot_logs_table.php` — `string('message')` → `text('message')`
